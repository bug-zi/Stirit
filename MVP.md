# 《你来说我来炒》MVP 设计文档

## 1. 项目定位

《你来说我来炒》是一款面向黑客松演示的快节奏单机料理小游戏。玩家经营一家未来校园夜市里的怪菜摊，根据顾客抽象、情绪化、带梗的需求，在限时内选择食材、调料和烹饪方式，做出一道怪菜。出锅后，AI 根据顾客需求和玩家组合进行评价，生成菜名、四维评分、评级和吐槽。

MVP 的目标不是做复杂经营系统，而是验证一个核心体验：

> 玩家用食材组合表达理解，AI 评价这道菜是否真正读懂了顾客。

## 2. MVP 核心体验

### 2.1 一句话玩法

顾客说不清自己想吃什么，玩家要在 60 秒内用有限选项做出最贴合需求的怪菜。

### 2.2 单局结构

- 玩家初始资金为 666。
- 每 5 单为 1 天，天末进入日结；游戏不再固定 5 单结束。
- 每单限时 60 秒。
- 每单随机抽取 1 位顾客订单。
- 每天第 5 单优先抽取高难度 Boss 订单。
- 每接待 1 位顾客都需要支付接待成本，成本随已完成订单数和顾客难度上升。
- 资金不足以接待下一位顾客时破产，进入最终结算。

### 2.3 每单流程

1. 顾客出现，显示身份和需求。
2. 系统展示本单接待成本和当前资金。
3. 玩家阅读需求。
4. 玩家选择 2-3 个食材。
5. 玩家选择 1-2 个调料。
6. 玩家选择 1 种烹饪方式。
7. 玩家点击“出锅”，或时间结束自动出锅。
8. 系统整理菜品语义摘要。
9. AI 生成评价结果。
10. 根据评级结算报酬、接待成本、净收益和经验。
11. 若经验达标则升级并选择 1 个增益 Buff。
12. 资金足够则继续下一单；资金不足则破产结算。

## 3. 游戏背景与设定

### 3.1 世界观

未来校园旁边有一条深夜美食街，其中有一家实验性小摊“怪菜快炒铺”。这家店没有固定菜单，专门服务说不清自己想吃什么的顾客。

顾客不会点“番茄炒蛋”，而是会提出类似这样的需求：

- “我想吃一个像期末周一样崩溃但还能续命的东西。”
- “给我来点像周五下午一样自由，但周一早上也能解释得过去的。”
- “我刚答辩完，想吃点能让我重新做人。”

### 3.2 玩家身份

玩家是“怪菜快炒铺”的实习厨师。玩家的目标不是做正常菜，而是读懂顾客的精神状态，用有限食材做出最贴合需求的怪菜。

### 3.3 AI 角色

游戏中的 AI 是“点单评价系统”，暂定名为“小炒 AI-7”。

它承担三个职责：

- 评价菜品是否贴合需求。
- 给菜品起名。
- 用带一点毒舌和幽默感的语气给出反馈。

## 4. 核心玩法系统

### 4.1 制作结构

每单玩家必须完成以下选择：

| 类型     | 选择数量 | 作用                 |
| -------- | -------: | -------------------- |
| 食材     |   2-3 个 | 决定菜品主体和场景   |
| 调料     |   1-2 个 | 强化味觉、情绪和风险 |
| 烹饪方式 |     1 个 | 放大表达倾向         |

合法出锅条件：

- 食材数量在 2-3 个之间。
- 调料数量在 1-2 个之间。
- 已选择 1 种烹饪方式。

不合法时，“出锅”按钮不可用，或点击后提示缺少内容。

## 5. 顾客与订单系统

### 5.1 订单来源

MVP 不在局内实时生成订单。订单由 AI 赛前批量生成，人工审核后写入本地内容库。局内只从本地库随机抽取。

这样做的原因：

- 避免实时生成卡顿。
- 保证需求文本质量。
- 避免 AI 生成不合适内容。
- 方便控制难度曲线。

### 5.2 订单字段

每条 `CustomerOrder` 建议包含：

| 字段          | 说明                     |
| ------------- | ------------------------ |
| id            | 唯一编号                 |
| customer_name | 顾客名                   |
| role          | 顾客身份                 |
| avatar_key    | 顾客头像或表情资源 key   |
| request       | 顾客需求文本             |
| difficulty    | 难度，1-5                |
| desired_tags  | 推荐命中的语义属性       |
| avoid_tags    | 不希望过多出现的语义属性 |
| boss          | 是否为 Boss 单           |

### 5.3 MVP 订单数量

- 普通订单：25 条。
- Boss 订单：5 条。
- 总计：30 条。

每 5 单为 1 天：

- 前 4 单：从普通订单中随机抽取，尽量不重复。
- 第 5 单：从 Boss 订单中随机抽取。

## 6. AI 评价系统

### 6.1 AI 评价目标

AI 不决定玩家是否能出锅，也不决定核心流程。AI 只在玩家出锅后进行评价，用自然语言理解顾客需求和菜品语义。

AI 需要输出：

- 菜名。
- 四维评分。
- 总评级。
- 简短评价。
- 顾客反应。

### 6.2 评价输入

提交给 AI 的信息应尽量结构化：

```json
{
  "customer": {
    "name": "赶项目的学生",
    "role": "熬夜写代码的本科生",
    "request": "我想吃一个像期末周一样崩溃但还能续命的东西。",
    "desired_tags": ["续命", "熬夜", "焦虑", "刺激"],
    "avoid_tags": ["过度健康", "太平淡"]
  },
  "dish": {
    "ingredients": ["泡面", "咖啡冻", "鸡蛋"],
    "seasonings": ["辣椒", "糖"],
    "cooking_method": "爆炒",
    "semantic_tags": ["便宜", "学生", "深夜", "续命", "提神", "焦虑", "营养", "刺激", "甜", "混乱"]
  },
  "system_score": {
    "relevance": 84,
    "taste": 66,
    "emotion": 88,
    "risk": 72
  }
}
```

### 6.3 评价输出

AI 必须返回固定 JSON：

```json
{
  "dish_name": "期末续命爆炒面",
  "scores": {
    "relevance": 88,
    "taste": 62,
    "emotion": 91,
    "risk": 74
  },
  "grade": "A",
  "comment": "这道菜很像凌晨三点的自救：不健康，但确实能把人从地上拽起来。",
  "customer_reaction": "顾客沉默三秒，然后开始继续写项目。"
}
```

### 6.4 四维评分定义

| 维度             |  范围 | 说明                               |
| ---------------- | ----: | ---------------------------------- |
| 贴题度 relevance | 0-100 | 是否回应了顾客真正想要的东西       |
| 美味度 taste     | 0-100 | 组合是否像能入口，是否味觉合理     |
| 情绪命中 emotion | 0-100 | 是否命中了需求中的情绪、场景和梗   |
| 风险值 risk      | 0-100 | 离谱、副作用、后悔概率，越高越危险 |

风险值不等同于扣分。怪菜可以有一定风险，但风险过高会影响最终等级。

### 6.5 推荐总分公式

系统先计算基础分，再由 AI 在合理范围内修正。

基础总分：

```text
base_total = relevance * 0.4 + taste * 0.2 + emotion * 0.3 + (100 - abs(risk - 55)) * 0.1
```

说明：

- 贴题度最重要。
- 情绪命中第二重要。
- 美味度保持基本约束。
- 风险值接近 55 最理想，太安全不怪，太危险像事故。

AI 最终评分允许在系统基础分上下浮动 15 分，但最终分数必须限制在 0-100。

### 6.6 等级标准

| 等级 | 分数范围 | 名称               |
| ---- | -------: | ------------------ |
| S    |   90-100 | 神之一锅           |
| A    |    80-89 | 精准怪菜           |
| B    |    70-79 | 勉强对味           |
| C    |    60-69 | 精神稳定但不太好吃 |
| D    |    45-59 | 顾客沉默           |
| F    |     0-44 | 厨房事故           |

## 7. 反馈与成长

### 7.1 单单反馈

每单结算展示：

- AI 菜名。
- 四维评分条。
- 总评级。
- 本单报酬、接待成本、净收益和当前资金。
- 本单经验奖励。
- 顾客表情变化。
- AI 点评文本。
- “下一单”按钮。

### 7.2 资金与破产

资金是游戏的主要生存资源。玩家初始资金为 666，每单根据顾客难度和已完成订单数计算接待成本：

```text
接待成本 = 50 + 已完成订单数 × 12 + 顾客难度成本
```

顾客难度成本：

| 难度 | 成本加成 |
| ---- | -------: |
| 1    |       +0 |
| 2    |      +20 |
| 3    |      +45 |
| 4    |      +60 |
| 5    |      +80 |

AI 评价越高，报酬越高；低评价会明显亏钱。

| 评级 | 报酬规则              | 经验 |
| ---- | --------------------- | ---: |
| S    | 成本 × 2.4 + 60       |   30 |
| A    | 成本 × 1.8 + 30       |   22 |
| B    | 成本 × 1.25           |   15 |
| C    | 成本 × 0.75           |    8 |
| D    | 成本 × 0.35           |    3 |
| F    | 成本 × 0.2            |    1 |

净收益：

```text
净收益 = 报酬 - 接待成本
```

若当前资金不足以支付下一位顾客的接待成本，则游戏结束并进入破产结算。

### 7.3 经验、头衔与 Buff

每单完成后根据评级获得经验。经验达到门槛时玩家升级，头衔变化，并从 3 个随机 Buff 中选择 1 个。

| 等级 | 头衔        | 累计经验 |
| ---- | ----------- | -------: |
| Lv.1 | 街边新厨    |        0 |
| Lv.2 | 稳锅学徒    |       40 |
| Lv.3 | 风味厨师    |       90 |
| Lv.4 | 人气主厨    |      160 |
| Lv.5 | AI 食神候补 |      250 |

MVP Buff 池：

- 好评小费：A/S 评价额外获得 20 金币。
- 成本控制：后续接待成本降低 10%。
- 招牌溢价：S 级菜品报酬提高 30%。
- 失误补救：第一次 D/F 评价按 C 级报酬结算。
- 连击热锅：A/S 连击报酬倍率额外提高 5%。

### 7.4 连击

连续获得 A 或 S 时触发连击：

- 连击 2：报酬 +10%。
- 连击 3：报酬 +20%。
- 连击 4 以上：报酬 +30%。

### 7.5 解锁

MVP 使用轻量解锁：

- 初始开放 8 个食材、5 个调料、4 种烹饪方式。
- 每局结束如果总评级达到 B 以上，解锁 1 个新食材或调料。
- 若实现时间不足，可改为所有内容默认解锁。

### 7.6 日结与破产结算

每 5 单结束后进入日结页，展示：

- 总分。
- 平均分。
- 最高单评分。
- 最佳菜名。
- 当前资金。
- 当前头衔。
- 累计报酬。

资金不足以接待下一位顾客时进入破产结算，展示：

- 接待顾客数。
- 撑到第几天。
- 最终资金。
- 最高头衔。
- 平均分。
- 最佳菜名。
- AI 总评称号。

称号示例：

- 期末续命大师。
- 灵魂爆炒专家。
- 夜市语义厨神。
- 厨房事故艺术家。
- 情绪调味师。

## 8. UI 与交互

### 8.1 主界面布局

建议单屏完成全部操作：

- 左侧：食材与调料列表，按分类分组。
- 中间：锅和已选择内容。
- 右侧：顾客头像、身份、需求文本、倒计时。
- 底部：烹饪方式按钮和“出锅”按钮。
- 顶部：当前天数、当前单数、资金、接待成本、头衔、经验。

### 8.2 操作方式

MVP 优先使用点击交互，降低实现风险：

- 点击食材或调料加入锅。
- 点击锅内项目移除。
- 点击烹饪方式按钮选择。
- 点击“出锅”提交。

如果时间充足，再补充拖拽交互。

### 8.3 状态提示

- 选择不足时，“出锅”按钮置灰。
- 选择已满时，继续点击同类选项给出提示。
- 时间低于 10 秒时，倒计时变色并播放紧张音效。
- AI 评价中显示“点单系统正在分析这锅东西”。
- AI 失败时显示“点单系统短路，启用本地评分”。

## 9. MVP 范围

### 9.1 必须包含

- 初始资金 666，并以破产作为游戏结束条件。
- 每 5 单为 1 天，并有日结页。
- 30 条本地订单。
- 食材、调料、烹饪方式选择。
- 合法出锅校验。
- AI 评价接口。
- AI 失败备用评分。
- 单单评分展示。
- 资金、成本、报酬、经验、升级 Buff。
- 破产最终结算页。

### 9.2 可以延后

- 店铺装修。
- 复杂剧情。
- 图片生成菜品。
- 语音播报。
- 在线排行榜。
- 存档系统。
- 多结局经营。
- 自定义订单编辑器。

## 10. MVP 验收标准

- 玩家能从开始页进入一局游戏。
- 玩家能完成至少 5 单并看到日结。
- 每单都能选择合法组合并出锅。
- 不合法组合不能提交。
- AI 成功时显示 AI 菜名、评分和吐槽。
- AI 失败时游戏不中断，并显示备用评价。
- 第 5 单能抽到 Boss 订单。
- 评分等级、报酬、接待成本、净收益、资金和经验能正确累计。
- 经验达标后能进入升级界面并选择 Buff。
- 资金不足以接待下一单时进入破产结算。
- 游戏能在无联网或 AI 不可用时完成完整经营流程。
