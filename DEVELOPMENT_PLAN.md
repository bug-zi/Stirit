# 《我来说你来炒》项目开发计划

## 1. 开发目标

在 26 小时内完成一款可演示的 Godot 4.6 单机 MVP：

- 玩家可以完成一局 5 单怪菜挑战。
- 局内订单来自本地审核内容库。
- 玩家通过“食材 + 调料 + 烹饪方式”制作怪菜。
- 出锅后调用 AI 生成评价。
- AI 不可用时使用本地备用评分。
- 游戏具备完整开始、游玩、单单评价、最终结算流程。

## 2. 团队分工

建议三人分工如下。

### 2.1 Godot 玩法负责人

负责：

- 项目结构和主场景。
- 游戏状态机。
- 5 单流程。
- 选择逻辑和合法性校验。
- 倒计时、提交、结算。
- 本地备用评分。

主要交付：

- 可完整游玩的单机流程。
- 与 AI 接口对接的数据输入和结果消费。

### 2.2 AI 与数据负责人

负责：

- 订单内容库。
- 食材、调料、烹饪方式数据。
- AI 评价 prompt。
- AI 返回 JSON 校验规则。
- 失败兜底逻辑的接口约定。

主要交付：

- 30 条审核订单。
- 20+ 个可选制作项。
- 稳定的 AI 评价输入输出格式。

### 2.3 UI/美术/反馈负责人

负责：

- 单屏 UI 布局。
- 食材、调料、烹饪方式按钮样式。
- 顾客头像和表情。
- 评分条、评级、金币、连击反馈。
- 出锅和结算动画。
- 音效和整体视觉风格。

主要交付：

- 可演示的完整界面。
- 明确、爽快、好笑的反馈体验。

## 3. 里程碑排期

以下按 26 小时黑客松节奏设计。

### 0-2 小时：项目定型

目标：所有人对范围达成一致，避免后期发散。

任务：

- 锁定玩法规则：5 单一局、每单 60 秒。
- 锁定选择规则：2-3 食材、1-2 调料、1 烹饪方式。
- 锁定评分维度：贴题度、美味度、情绪命中、风险值。
- 创建 Godot 项目，配置 `main.tscn` 为主场景。
- 确定数据文件格式和字段。

验收：

- 能打开项目主场景。
- 玩法负责人、AI 负责人、UI 负责人使用同一份数据字段。

### 2-6 小时：核心流程原型

目标：先让游戏“能跑完一局”。

任务：

- 搭建主界面占位布局。
- 实现订单抽取。
- 实现选择食材、调料、烹饪方式。
- 实现合法出锅校验。
- 实现每单倒计时。
- 实现 5 单推进和最终结算。
- 使用本地假评分完成流程。

验收：

- 不接 AI 也能完成一局。
- 能看到每单结果和最终总分。
- 时间到能自动提交或进入结算。

### 6-10 小时：内容库与基础评分

目标：让本地数据足够支撑试玩。

任务：

- 整理 30 条顾客订单。
- 整理 12 个食材、8 个调料、4 种烹饪方式。
- 为制作项配置后台语义属性。
- 实现标签汇总。
- 实现系统基础评分。
- 实现 S/A/B/C/D/F 等级。

验收：

- 每局前 4 单从普通订单随机抽取。
- 第 5 单从 Boss 订单随机抽取。
- 相同选择能稳定得到相近基础分。
- 明显贴题的组合分数高于明显不贴题的组合。

### 10-15 小时：AI 评价接入

目标：让 AI 成为可演示核心亮点。

任务：

- 组装 AI 输入 JSON。
- 编写评价 prompt。
- 调用 AI 接口。
- 解析 AI JSON 输出。
- 校验 `dish_name`、`scores`、`grade`、`comment`、`customer_reaction`。
- 实现 AI 超时、空文本、非法 JSON 的备用评分。

验收：

- AI 成功时能显示菜名、四维评分和吐槽。
- AI 失败时游戏不中断。
- AI 评价文本与顾客需求和菜品组合有关。
- AI 输出不合法时不会导致 Godot 报错。

### 15-20 小时：反馈与可玩性打磨

目标：让游戏从“能玩”变成“想再玩一局”。

任务：

- 增加评分条动画。
- 增加顾客表情变化。
- 增加出锅反馈。
- 增加 A/S 连击。
- 增加金币奖励。
- 增加最终称号。
- 调整订单难度和评分范围。

验收：

- 玩家每次出锅后能快速理解自己为什么得分高或低。
- 连续高分有明显奖励反馈。
- 第 5 单有 Boss 感。
- 试玩者能在第一局内理解玩法。

### 20-24 小时：内容扩充与演示准备

目标：准备可稳定展示的版本。

任务：

- 检查 30 条订单质量。
- 补充顾客头像或占位头像。
- 补充按钮图标和音效。
- 调整 UI 文案。
- 准备 2-3 个演示用高质量订单。
- 准备 AI 失败时的演示兜底。

验收：

- 连续游玩 3 局没有流程中断。
- 演示订单能稳定产生有趣评价。
- 无 AI 网络时仍可完整演示。

### 24-26 小时：冻结与最终测试

目标：停止加功能，只修阻断问题。

任务：

- 运行 Godot headless 校验。
- 完整试玩至少 3 轮。
- 修复会阻断演示的问题。
- 确认主场景配置正确。
- 准备演示话术和玩法说明。

验收：

- 开始到结算流程稳定。
- AI 成功路径稳定。
- AI 失败路径稳定。
- 项目能在演示机器上打开运行。

## 4. 数据设计

### 4.1 CustomerOrder

```json
{
  "id": "order_001",
  "customer_name": "阿困",
  "role": "熬夜赶项目的学生",
  "avatar_key": "student_sleepy",
  "request": "我想吃一个像期末周一样崩溃但还能续命的东西。",
  "difficulty": 2,
  "desired_tags": ["续命", "熬夜", "焦虑", "刺激"],
  "avoid_tags": ["太平淡", "过度健康"],
  "boss": false
}
```

### 4.2 FoodItem

食材和调料可以共用结构，用 `category` 区分。

```json
{
  "id": "ingredient_noodle",
  "name": "泡面",
  "category": "ingredient",
  "tags": ["便宜", "学生", "深夜", "续命", "敷衍"],
  "description": "深夜救命粮，便宜但很有故事。",
  "unlocked": true
}
```

### 4.3 CookingMethod

```json
{
  "id": "method_stir_fry",
  "name": "爆炒",
  "tags": ["刺激", "热烈", "混乱", "上头"],
  "description": "把所有情绪都炒到冒烟。",
  "effect_key": "hot"
}
```

### 4.4 EvaluationResult

```json
{
  "dish_name": "期末续命爆炒面",
  "scores": {
    "relevance": 88,
    "taste": 62,
    "emotion": 91,
    "risk": 74
  },
  "grade": "A",
  "comment": "这道菜很像凌晨三点的自救：不健康，但确实能把人从地上拽起来。",
  "customer_reaction": "顾客沉默三秒，然后开始继续写项目。"
}
```

## 5. AI 接口约定

### 5.1 AI 请求内容

AI 请求必须包含：

- 顾客身份。
- 顾客需求。
- 推荐命中标签。
- 避免标签。
- 玩家选择的食材。
- 玩家选择的调料。
- 玩家选择的烹饪方式。
- 系统汇总语义属性。
- 系统基础评分。

### 5.2 AI 输出约束

AI 只能输出 JSON，不输出额外解释文本。

必须包含字段：

- `dish_name`
- `scores.relevance`
- `scores.taste`
- `scores.emotion`
- `scores.risk`
- `grade`
- `comment`
- `customer_reaction`

校验规则：

- 所有评分必须是 0-100 的整数。
- `grade` 必须是 `S/A/B/C/D/F` 之一。
- 文本字段不能为空。
- 如果校验失败，使用本地备用评价。

### 5.3 超时策略

- AI 请求超时时间建议为 6 秒。
- 超时后立刻使用本地备用评分。
- UI 显示“点单系统短路，启用本地评分”。
- 不重试，避免阻塞游戏节奏。

## 6. 备用评分策略

备用评分用于 AI 不可用时保证完整游戏流程。

### 6.1 贴题度

根据玩家菜品标签命中 `desired_tags` 的比例计算。

### 6.2 美味度

根据食材和调料的组合合理性计算。MVP 可使用简单规则：

- 至少 1 个稳定食材，如米饭、鸡蛋、泡面，增加美味度。
- 强冲击调料超过 2 个，降低美味度。
- 甜、辣、苦、清凉同时出现过多，降低美味度。

### 6.3 情绪命中

根据需求关键词和菜品标签重合度计算。

### 6.4 风险值

根据刺激、危险、上头、争议、罪恶等标签数量计算。

## 7. 测试清单

### 7.1 流程测试

- 开始游戏后能进入第 1 单。
- 每单能正常倒计时。
- 手动出锅能进入评价页。
- 时间结束能自动提交。
- 第 5 单结束后能进入结算页。

### 7.2 选择测试

- 少于 2 个食材不能提交。
- 多于 3 个食材不能继续添加。
- 少于 1 个调料不能提交。
- 多于 2 个调料不能继续添加。
- 未选择烹饪方式不能提交。
- 点击已选项目可以移除。

### 7.3 数据测试

- 普通订单不会在同一局重复。
- Boss 单只出现在第 5 单。
- 食材、调料、烹饪方式能正确汇总标签。
- 解锁字段不会导致已锁内容误显示。

### 7.4 AI 测试

- AI 返回合法 JSON 时正常展示。
- AI 返回非法 JSON 时启用备用评分。
- AI 超时时启用备用评分。
- AI 返回评分越界时被限制到 0-100。
- AI 返回空文本时不影响流程。

### 7.5 结算测试

- 总分正确累计。
- 平均分正确计算。
- 最高单正确记录。
- 最佳菜名正确显示。
- 最终称号正确生成。

## 8. 风险与降级方案

| 风险 | 影响 | 降级方案 |
|---|---|---|
| AI 接口不稳定 | 评价卡住 | 6 秒超时，使用本地备用评分 |
| 内容库不够有趣 | 玩法显得平 | 优先打磨 10 条演示订单 |
| UI 拖拽耗时 | 操作实现风险 | MVP 使用点击选择 |
| 评分不稳定 | 玩家不理解结果 | 代码基础分占主导，AI 只做修正和解释 |
| 26 小时不够 | 功能未完成 | 放弃解锁和动画，保留完整 5 单流程 |

## 9. 最终交付标准

交付版本必须满足：

- 可启动。
- 可完成一局。
- 可展示 AI 评价。
- AI 不可用仍可完整游玩。
- 内容足够支撑 3-5 分钟演示。
- 不包含在线多人、复杂经营、图片生成、语音播报等非 MVP 功能。
